# prompt-injection-lab — 実装計画書 (Phase 2)

> 本計画は DESIGN §4.1（制御反転・OpenAI 互換シム）・§4.2（EnvKind vs adapter）・§8.3（fine-tuned judge）に依拠する．

## 0. 成功条件（Definition of Done — Phase 2）

Phase 1 の DoD に加え，以下が CLI から再現可能であること：

1. **OpenAI 互換シム**：Rust プロセスが OpenAI 互換のローカルエンドポイントを立て，`base_url` 差し替えで外部（Python）クライアントの生成要求を **`pil-llm` の単一経路**に集約できる（温度・seed・cache・metadata・rate-limit が全経路で揃う）．
2. **sidecar adapter**：Python 上流ベンチ本体（環境・ツール実装のみ）を薄い殻として起動し，モデル呼び出しは全てシム経由で Rust に戻る（native-first，§4.1）．
3. **環境種別を跨いだ比較の成立**：`EnvKind::StaticPrompt`（Phase 1）と `EnvKind::Emulated`（AgentDojo）が併存し，集計 API が「同一 EnvKind 内でのみ比較」を型で強制しつつ，明示 opt-in で跨ぎ比較（Kendall's W 等）を開示する．
4. **fine-tuned judge**：`qylu4156/strongreject-15k-v1`（gemma-2b LoRA）の logit 期待値式判定を `instrument` として追加し，`reliability` で他 judge と突き合わせできる．
5. **network-free 既定の維持**：シム・sidecar・実 LLM 経路は全て cargo feature で gate し，既定ビルド・既定テストはネットワーク非依存のまま（loopback + MockProvider でシムを検証）．

## 2. マイルストーン詳細

### P2-M1 — `pil-shim`（OpenAI 互換ローカルエンドポイント）★keystone / §4.1

- 新 crate `crates/pil-shim`．`Arc<dyn LlmProvider>`（= `pil-llm` の単一経路）を背後に，OpenAI `/v1/chat/completions`（+ `/v1/models`）を localhost で提供する．
- 変換：OpenAI ChatCompletion 要求 → `GenerateRequest`（`messages`→rendered prompt / `temperature`・`seed`・`max_tokens`・system→`LlmConfig` / `logprobs`・`top_logprobs`→§6.3）→ provider 呼び出し → `GenerateOutput` → OpenAI ChatCompletion 応答（`CallMetadata` を全件記録）．
- feature `shim` で HTTP サーバ依存（axum/hyper on tokio．既定は無効）を gate．
- テスト（network-free）：エフェメラルポートで起動し，loopback にプレーン HTTP で chat completion を POST → `MockProvider` に到達したこと・応答スキーマ・metadata 記録を検証．logprobs passthrough / エラーマッピングも単体化．
- 完了条件：既定ビルドは reqwest/axum 非依存のまま，`--features pil-shim/shim` でシムがビルド・起動・loopback テスト緑．

### P2-M2 — tool-calling スキーマ変換（OpenAI ↔ Anthropic/Gemini）§4.1

- OpenAI の tool schema / `tool_calls` を Anthropic・Gemini 形式へ相互変換する純関数群．シムが単一スキーマで任意バックエンドを提供できるようにする．
- テスト：各方向の固定フィクスチャで round-trip・欠損フィールドの扱いを検証．
- 注：AgentDojo の対象タスクが tool-calling を要する範囲に限定して実装（過剰実装を避ける）．

### P2-M3 — sidecar adapter 基盤（`pil-sidecar`）§4.1 native-first

- Python sidecar プロセスを起動し，`OPENAI_BASE_URL`/`OPENAI_API_KEY` をシムに向ける．ライフサイクル・IPC・入出力正規化・provenance 記録は全て Rust．Python は irreducible な環境/ツール本体のみ．
- feature gate．テストは極小スタブ Python（シム loopback を叩くだけ）で network-free に検証．
- 判断基準（§4.1）：「Rust に移すと測定値が変わりうるか」— 変わるなら Python 温存，変わらないなら Rust．

### P2-M4 — AgentDojo 取り込み（`EnvKind::Emulated` 初の実データ）§10

- AgentDojo を `third_party/` に submodule 追加（SHA 固定）．`pil-bench-agentdojo`（sidecar adapter；74 tools / 97 tasks / 629 security cases）を env=`Emulated` で実装．シム経由で配線．
- AgentDojo は `openai-compatible` プロバイダを持ちモンキーパッチ不要（§4.1）．`base_url` 差し替え可否を実測で確認．

### P2-M5 — fine-tuned judge（`qylu4156/strongreject-15k-v1`）§8.3

- LoRA を merge→GGUF 化し Ollama で動かす（または vLLM sidecar）．`top_logprobs`（Ollama v0.12.11+）でトークン `1`〜`5` の分布を取り，`softmax · linspace(0,1,5)` の期待値をスコアとする `instrument` を `pil-metrics` に追加．
- sidecar トラックと独立．`reliability` に合流し他 judge と突き合わせ．

### P2-M6 — 横断比較レポート（cross-EnvKind）§2.3 / §10

- StaticPrompt（Phase 1）と Emulated（AgentDojo）併存下で，「同一 EnvKind 内でのみ比較」を強制しつつ，明示 opt-in（`CrossEnv` マーカー，Phase 1 で private field 導入済み）で跨ぎ比較を開示．ベンチ/ランキング横断の Kendall's W を出す．
- `pil-report` + `pil-cli` を拡張．

### P2-M7 — E2E（Phase 2 総合）

- CLI で混在 suite（StaticPrompt + Emulated）を実行し，report が跨ぎ比較の拒否/opt-in・fine-tuned judge を含む reliability を再現．§0 DoD 全項目を CLI から確認．

## 3. テスト戦略

- 既定テストは network-free 維持（シムは loopback + MockProvider，sidecar はスタブ Python）．
- 実 LLM / 実 AgentDojo を要するテストは feature gate + `#[ignore]` で分離し，CI 既定から外す．
- Phase 1 の回帰（175 tests）を壊さないことを各 M で確認．

## 4. 技術スタック（推奨・確定は実装時）

- HTTP サーバ：axum（tokio 既存，hyper ベース）— feature `shim` でのみ導入．
- sidecar：`std::process` + `tokio` プロセス制御，環境変数注入．
- fine-tuned judge：GGUF 化（`llama.cpp` 系ツールでの merge/convert）は事前手順とし，Rust 側は logprob 期待値式のみ持つ．

## 5. リスクと残タスク

- **シムの logprobs 実返却**：Ollama OpenAI 互換経路での logprobs 実返却は未実測（Phase 1 §6.3 で native は確定）．P2-M1/M5 で実測する．
- **AgentDojo の Python 依存範囲**：どこまで Rust に剥がせるかは AgentDojo のコードを読んで個別判定（§4.1）．
- **GGUF 化の実現性**：LoRA merge→GGUF が Ollama で `top_logprobs` を返すか要検証．駄目なら vLLM sidecar に退避．
